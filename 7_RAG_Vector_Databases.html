RAG Basics — Retrieval-Augmented Generation

LLM + Search Engine = RAG. Yeh combination kyun powerful hai?

1.1 RAG Kya Hota Hai?

RAG ek AI architecture pattern hai jisme Language Model ko answer generate karne se pehle relevant documents retrieve karke dikhaye jaate hain.

Simple analogy: Maan lo tum ek exam de rahe ho. Ek taraf Closed-book exam hota hai (sirf memory pe rely karo) — yahi Pure LLM hai. Doosri taraf Open-book exam hota hai (reference material dekh sakte ho) — yahi RAG hai!

📖 Definition

RAG = Retrieval-Augmented Generation
"Pehle dhundho (Retrieve), phir context ke saath answer banao (Generate)" — yahi RAG ka core idea hai. Yeh 2020 mein Meta AI researchers ne introduce kiya tha.

1.2 RAG Kyun Chahiye? — Problem Statement
LLMs Ki Limitations
🕐 Knowledge Cutoff
GPT-4 ka training data ek date tak hai. Uske baad ki news, events, company data — kuch bhi nahi pata.
🏢 Private Data Problem
Tumhare company ke internal documents, HR policies, product manuals — LLM ne kabhi nahi dekhe.
👻 Hallucination
LLM confident hokar galat information deta hai. Bina source ke koi accountability nahi.
💰 Cost of Fine-tuning
Naya data sikhane ke liye fine-tuning bahut expensive aur time-consuming hai. RAG much cheaper hai.
RAG Ke Benefits
✅ RAG Solves These
  • 🔄 Dynamic Knowledge: Naya document add karo, AI turant updated ho jaata hai
  • 📚 Source Citations: AI bata sakta hai "yeh answer XYZ document se liya"
  • 🏢 Private Data: Company ke internal docs safely use ho sakte hain
  • 💸 Cost Effective: Fine-tuning se 10-100x sasta
  • ✅ Hallucination Reduce: Grounded responses with real context

1.4 Step-by-Step RAG Walkthrough
1
Document Ingestion Tumhare paas ek 500-page company manual hai. Usse system mein load karo — PDF reader, web scraper, ya database se data nikalo.
2
Chunking 500 pages ek saath LLM ko dena possible nahi (context window limit!). Isliye document ko chhote-chhote chunks mein todo — har chunk ~200-500 words ka.
3
Embedding Generation Har chunk ko ek embedding model (jaise text-embedding-ada-002 ya all-MiniLM) se pass karo. Output milega ek vector — numbers ki ek list jo us chunk ka "meaning" represent karti hai.
4
Vector Database Storage Ye vectors ek special database (Pinecone, ChromaDB, etc.) mein store ho jaate hain jo fast similarity search kar sakta hai.
5
Query Processing User ka question bhi embedding model se pass karo — ek query vector banta hai.
6
Similarity Search Query vector ko database ke saare vectors se compare karo. Jo vectors sabse "similar" hain (mathematically close) — unhe retrieve karo. Yeh typically Top-K results hote hain (jaise top 5 chunks).
7
Context Injection Retrieved chunks ko LLM ke prompt mein inject karo: "Niche diye gaye documents ke basis par sawal ka jawab do: [CONTEXT] ... [QUESTION]"
8
Answer Generation LLM ab grounded context ke saath accurate, cited answer generate karta hai. Hallucination drastically kam ho jaati hai!
1.5 Basic RAG — Python Code
simple_rag.py — Basic RAG System
PYTHON
# =====================================================
# PART 7: Basic RAG System — Beginners Ke Liye
# RAG = Documents Load → Embed → Store → Retrieve → Generate
# =====================================================

from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# ─── STEP 1: Documents Load Karo ───────────────────
print("📄 Documents load kar raha hoon...")
loader = PyPDFLoader("company_manual.pdf")
documents = loader.load()
print(f"Total pages: {len(documents)}")

# ─── STEP 2: Chunking — Chhote Pieces Banao ────────
# chunk_size = har chunk mein kitne characters
# chunk_overlap = consecutive chunks mein overlap (context loss avoid karne ke liye)
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", ".", " "]  # Yahan se todna prefer karta hai
)
chunks = splitter.split_documents(documents)
print(f"Total chunks bane: {len(chunks)}")

# ─── STEP 3: Embeddings Model Initialize Karo ──────
embeddings = OpenAIEmbeddings(
    model="text-embedding-3-small"  # Fast aur cheap
)

# ─── STEP 4: Vector Store Mein Save Karo ───────────
print("🔢 Embeddings bana raha hoon aur store kar raha hoon...")
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"  # Disk pe save ho jaayega
)

# ─── STEP 5: Retriever Banao ────────────────────────
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4}  # Top 4 similar chunks lao
)

# ─── STEP 6: LLM + RAG Chain Banao ─────────────────
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

rag_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True  # Sources bhi batao!
)

# ─── STEP 7: Query Karo ─────────────────────────────
question = "Leave policy kya hai company mein?"
result = rag_chain.invoke({"query": question})

print("\n🤖 Answer:", result["result"])
print("\n📚 Sources Used:")
for doc in result["source_documents"]:
    print(f"  - Page {doc.metadata['page']}: {doc.page_content[:100]}...")
▶ OUTPUT

Total pages: 47
Total chunks bane: 312
🔢 Embeddings bana raha hoon aur store kar raha hoon...

Embeddings & Vector Space

Text ko numbers mein convert karna — AI ka secret language

2.1 Embedding Kya Hota Hai?

Embedding ek mathematical representation hai jo words, sentences, ya documents ko numbers ki list (vector) mein convert karta hai — is tarah ki similar meanings wale texts ke vectors paas-paas hote hain space mein.

Simple analogy: Soch lo ek map hai jahan:

  • "King" aur "Queen" kaafi paas hain (royalty)
  • "Dog" aur "Cat" paas hain (pets)
  • "King" aur "Dog" dur hain (alag domains)
  • "Puppy" aur "Dog" bahut paas hain (almost same meaning)

Iska matlab hai har text ke liye 1536 numbers generate hote hain. Ye numbers us text ka "meaning" capture karte hain. Zyaada dimensions = better quality, lekin zyaada storage/computation.

2.2 Similarity Search — Similar Texts Kaise Dhundhen?

Do vectors kitne "similar" hain — yeh measure karne ke liye similarity metrics use hoti hain:

MetricFull NameRangeBest ForFormula
Cosine Similarity Angle between vectors -1 to 1 Text Similarity A·B / (|A| × |B|)
Dot Product Inner product -∞ to ∞ Normalized vectors Σ(Aᵢ × Bᵢ)
Euclidean Distance L2 Distance 0 to ∞ Dense vectors √Σ(Aᵢ-Bᵢ)²
Manhattan Distance L1 Distance 0 to ∞ Sparse data Σ|Aᵢ-Bᵢ|
2.3 Cosine Similarity — Sabse Popular Metric
cos(θ) = (A · B) / (|A| × |B|)
A · B = Dot product (multiply karke add karo)
|A|, |B| = Magnitudes (vector ka size)
θ = Angle between vectors
🎯 Interpretation

1.0 → Identical meaning
0.8+ → Very similar
0.5 → Somewhat similar
0.0 → Completely different
-1.0 → Opposite meaning

cosine_similarity.py — Manual Implementation
PYTHON
import numpy as np

# Cosine Similarity manually calculate karo
def cosine_similarity(vec_a, vec_b):
    """
    Do vectors ke beech cosine similarity nikalo.
    1.0 = same meaning, 0.0 = no relation, -1.0 = opposite
    """
    dot_product = np.dot(vec_a, vec_b)      # A · B
    mag_a = np.linalg.norm(vec_a)              # |A|
    mag_b = np.linalg.norm(vec_b)              # |B|
    return dot_product / (mag_a * mag_b)

# ─── Real embeddings ke saath example ──────────────
from openai import OpenAI

client = OpenAI()

def get_embedding(text):
    response = client.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding

# Teen sentences ke embeddings nikalo
sent1 = "Dog ek pet animal hai jo bahut loyal hota hai"
sent2 = "Kutte bahut wafadar jaanwar hote hain"  # Same meaning, different words
sent3 = "Python ek programming language hai"     # Completely different

emb1 = get_embedding(sent1)
emb2 = get_embedding(sent2)
emb3 = get_embedding(sent3)

# Similarities compare karo
print(f"sent1 vs sent2 (same meaning): {cosine_similarity(emb1, emb2):.4f}")
print(f"sent1 vs sent3 (different):   {cosine_similarity(emb1, emb3):.4f}")
print(f"sent2 vs sent3 (different):   {cosine_similarity(emb2, emb3):.4f}")

⚠️ Common Mistake

Kabhi bhi indexing aur querying mein alag embedding models use mat karo! Agar documents index karte waqt OpenAI use kiya, toh query karte waqt bhi OpenAI hi use karo. Different models ke vectors directly comparable nahi hote.

2.5 Semantic Search vs Keyword Search
🔤 Keyword Search (Traditional)
Query: "heart attack symptoms"

Will find: Documents containing exact words "heart attack symptoms"
Won't find: "myocardial infarction ke signs" — even though same meaning!

Query: "heart attack symptoms"

Vector Databases

Millions of vectors ko efficiently store aur search karna

3.1 Vector Database Kya Hai?

Ek Vector Database ek specialized database hai jo vectors (high-dimensional numbers) ko efficiently store karta hai aur similarity-based search karta hai — unlike traditional SQL databases jo exact match dhundti hain.

💡 Analogy — Library vs Google

Traditional DB (SQL) = Library mein exact book number se dhundna
Vector DB = Google pe "programming book for beginners" type karo — Google similar books suggest karta hai even if exact words match nahi hote!

3.2 Vector DB Ka Internal Working — ANN Index

Normal exact nearest neighbor search (Brute Force) pe millions of vectors scan karna bahut slow hoga — O(N × D) complexity. Isliye vector DBs use karte hain ANN (Approximate Nearest Neighbor) algorithms:

AlgorithmFull NameSpeedAccuracyMemory
HNSW Hierarchical Navigable Small World Very Fast High High RAM
IVF Inverted File Index Fast Medium Low
LSH Locality Sensitive Hashing Medium Medium Very Low
ScaNN Scalable Nearest Neighbors Very Fast High Medium
PQ Product Quantization Fast Lower Very Low
🏆 Industry Standard

HNSW (Hierarchical Navigable Small World) sabse popular algorithm hai production mein. Pinecone, Weaviate, aur Qdrant sab primarily HNSW use karte hain. Yeh logarithmic time O(log N) mein search karta hai!

3.3 HNSW — Kaise Kaam Karta Hai?
🕸️ HNSW GRAPH STRUCTURE — Layers

Search: Top layer pe start karo → Greedy search → Base layer pe final neighbors

3.4 Major Vector Databases — Deep Dive

🌲 Pinecone

Type: Managed Cloud | Best For: Production, Enterprise | Language: REST API

Kya hai: Pinecone ek fully managed, serverless vector database hai. Tumhe infrastructure ki tension nahi — bas API call karo aur kaam karo.

Key Features: Real-time updates, automatic scaling, hybrid search support, namespaces (different collections), metadata filtering.

Best For: Production RAG apps jahan uptime critical hai, startups jo infra manage nahi karna chahte.

pinecone_example.py
PYTHON
from pinecone import Pinecone, ServerlessSpec

# Initialize Pinecone
pc = Pinecone(api_key="your_api_key")

# Index banao (ek baar)
pc.create_index(
    name="company-docs",
    dimension=1536,          # OpenAI embedding size
    metric="cosine",
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)
index = pc.Index("company-docs")

# Vectors upsert karo
vectors = [
    ("doc1", [0.1, 0.2, ...], {"source": "HR_policy.pdf"}),
    ("doc2", [0.3, 0.4, ...], {"source": "Leave_rules.pdf"})
]
index.upsert(vectors=vectors)

# Query karo
results = index.query(
    vector=query_embedding,
    top_k=5,
    filter={"source": {"$eq": "HR_policy.pdf"}},  # Metadata filter!
    include_metadata=True
)

🎨 ChromaDB

Type: Open Source (Local + Cloud) | Best For: Development, Prototyping | Language: Python-native

Kya hai: ChromaDB sabse beginner-friendly vector database hai. Pure Python, koi setup nahi, ek line mein start ho jaata hai. Developer-first approach.

Key Features: Embedded mode (in-memory/disk), client-server mode, automatic embeddings (Chroma khud generate karta hai!), LangChain integration, persistent storage.

Best For: Beginners, prototyping, local development, Jupyter notebooks.

⚡ FAISS (Facebook AI Similarity Search)

Type: Open Source Library | Best For: High Performance, On-premise | Developer: Meta AI

Kya hai: FAISS ek library hai (database nahi), jo extremely fast vector search karta hai. Meta ne apne billion-scale use cases ke liye banaya. GPU support bhi hai!

Key Features: Blazing fast, GPU acceleration, multiple index types (Flat, IVF, HNSW, PQ), billion-scale vectors handle kar sakta hai.

Best For: Research, offline batch processing, extreme performance needs, on-premise deployment jahan cloud nahi use karna.

faiss_example.py
PYTHON
import faiss
import numpy as np

# ─── FAISS Index Banao ────────────────────────────
dimension = 1536  # OpenAI embedding size

# Flat L2 — Most accurate, slow for huge datasets
index_flat = faiss.IndexFlatL2(dimension)

# IVF + PQ — Fast aur compressed (large scale ke liye)
quantizer = faiss.IndexFlatL2(dimension)
index_ivf = faiss.IndexIVFFlat(quantizer, dimension, 100)  # 100 clusters

# ─── Random vectors add karo (real mein embeddings honge) ─
n_vectors = 10000
vectors = np.random.random((n_vectors, dimension)).astype('float32')

index_flat.add(vectors)
print(f"Total vectors indexed: {index_flat.ntotal}")

# ─── Query ────────────────────────────────────────
query = np.random.random((1, dimension)).astype('float32')

# k=5 nearest neighbors dhundo
distances, indices = index_flat.search(query, k=5)
print(f"Nearest neighbor indices: {indices[0]}")
print(f"Distances: {distances[0]}")

Key Features: Object-based storage, GraphQL API, built-in text vectorizers, image vectorizers, hybrid BM25+vector search, pluggable modules.

Chunking Strategies

Documents ko efficient pieces mein todna — RAG ki quality yahan decide hoti hai!

4.1 Chunking Kya Hai Aur Kyun Critical Hai?

Chunking ka matlab hai ek bade document ko chhote, manageable pieces mein todna jo vector database mein store ho sakein aur LLM context mein fit ho sakein.

⚠️ Chunking Quality = RAG Quality

Agar chunking galat hai, toh perfect embedding model bhi kaam nahi karega. Ye sab-se neglected lekin sab-se important step hai RAG pipeline mein. Production failures ka 60%+ reason poor chunking hota hai!

❌ Chunk Too Small

Chunk: "... temperature ko 0 rakhein ..."
Problem: Context missing! Kya temperature? Kis cheez ka? Retrieval useless ho jaata hai.

❌ Chunk Too Large

Chunk: Poora 10-page chapter ek saath
Problem: Specific answer dhundna mushkil, noise zyaada, LLM context waste.

4.4 Optimal Chunk Size — Kya Decide Karta Hai?
📏 Chunk Size Decision Factors
Small chunks (100-300 tokens): Precise retrieval, FAQ-style, specific facts

Medium chunks (300-600 tokens): Best for most use cases — balance of context aur precision

Large chunks (600-1000 tokens): Complex reasoning, when full context needed
🔁 Chunk Overlap Best Practices
Overlap = 10-20% of chunk size

500 char chunk → 50 char overlap
1000 token chunk → 100 token overlap

Zero overlap: Fast, lekin context break hone ka risk
Too much overlap: Redundant data, wasted tokens
🧠 Chunking Golden Rule

"Ek chunk = Ek complete thought"
Agar tumhara chunk ek complete, self-contained idea express karta hai jise padhke context samajh aaye — toh chunking correct hai!

💡 Production Tip — Parent-Child Chunking

Advanced technique: Index karne ke liye small chunks use karo (precise retrieval), but LLM ko unka parent (larger) chunk do (full context). Yeh "small-to-big retrieval" ya "parent document retriever" kehlaata hai aur retrieval quality dramatically improve karta hai.

  • Same chunk size har document ke liye — Legal doc aur FAQ alag strategy chahte hain
  • Zero overlap — Important sentences cut ho jaati hain
  • Tables/lists ko text ki tarah chunking — Tabular data structured chunks chahta hai
  • Metadata include nahi karna — Source, page number, date embedding ke saath store karo
  • Post-chunking quality check nahi karna — Random chunks manually review karo

    🔍 RETRIEVAL TECHNIQUES TAXONOMY
    5.2 Dense Retrieval — Semantic Search

    Kya hai: Query aur documents dono ko dense vectors (embeddings) mein convert karo, phir vector similarity search karo. Meaning-based matching hoti hai.

    ✅ Dense Retrieval Strengths
    • ✓ Semantic meaning understand karta hai
    • ✓ Paraphrasing aur synonyms handle karta hai
    • ✓ Cross-lingual search possible
    • ✓ Complex queries ke liye better
    ❌ Dense Retrieval Weaknesses
    • ✗ Exact keyword match miss kar sakta hai
    • ✗ Rare terms ke liye poor (product IDs)
    • ✗ Embedding model training data pe depend
    • ✗ Computation expensive
    5.3 Sparse Retrieval — BM25/TF-IDF

    Kya hai: Traditional information retrieval technique jo keyword frequency pe based hai. BM25 (Best Match 25) Elasticsearch ka default algorithm hai.

    📐 BM25 Formula Intuition

    TF (Term Frequency): Document mein word kitni baar aaya?
    IDF (Inverse Document Frequency): Word kitna rare hai across all docs?
    BM25 Score = TF normalization × IDF — Common words (the, is, a) ko penalize karta hai, rare meaningful words ko reward karta hai.

    ✅ Sparse Strengths
    • ✓ Exact keyword match excellent
    • ✓ Product codes, names, IDs ke liye perfect
    • ✓ Very fast, no GPU needed
    • ✓ Explainable results
    ❌ Sparse Weaknesses
    • ✗ Semantic meaning miss karta hai
    • ✗ Synonyms handle nahi karta
    • ✗ Typos pe fail hota hai
    • ✗ Cross-lingual search possible nahi
    5.4 Hybrid Search — Best of Both Worlds!

    Hybrid search Dense (semantic) + Sparse (keyword) dono ko combine karta hai. Results ko Reciprocal Rank Fusion (RRF) ya weighted scoring se merge karta hai.

    from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS

# Documents ki list

# ─── 1. Dense Retriever (Vector-based) ────────────
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(docs, embeddings)
dense_retriever = vectorstore.as_retriever(
    search_kwargs={"k": 10}
)

bm25_retriever.k = 10

# ─── 3. Ensemble = Hybrid Search! ─────────────────
hybrid_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, dense_retriever],
    weights=[0.4, 0.6],  # BM25: 40%, Dense: 60%
    # RRF (Reciprocal Rank Fusion) se merge hota hai automatically
)

# ─── Query ────────────────────────────────────────
results = hybrid_retriever.invoke("Python programming language")
print(f"Retrieved {len(results)} documents from hybrid search")

# ─── RRF Manual Implementation (Understanding ke liye) ──
def reciprocal_rank_fusion(result_lists, k=60):
    """
    k=60: Standard constant, results ko smooth karta hai
    Score = Σ(1 / (k + rank_i)) for each result list
    """
    scores = {}
    for results in result_lists:
        for rank, doc_id in enumerate(results):
            if doc_id not in scores:
                scores[doc_id] = 0
            scores[doc_id] += 1 / (k + rank + 1)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)

    6.4 Multi-Hop RAG — Complex Reasoning

    Kya hai: Ek question answer karne ke liye multiple retrieval steps chahiye. Har step ki output next step ka input banती hai.

    💡 Multi-Hop Example

    Question: "India ke Prime Minister ki alma mater kaunsa university hai?"
    Step 1: "India ke Prime Minister kaun hain?" → "Narendra Modi" retrieve karo
    Step 2: "Narendra Modi ki education" → University retrieve karo
    Answer: "Narendra Modi ne Gujarat University se Political Science padha"

    multi_hop_rag.py — Iterative Retrieval
    PYTHON
    from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate

# ─── Multi-Hop RAG — Iterative Retrieval ──────────
def multi_hop_retrieve(question, retriever, llm, max_hops=3):
    """
    Har hop mein:
    1. Current question se retrieve karo
    2. LLM se next question generate karo (ya final answer)
    3. Answer complete hua? Stop. Nahi? Agle hop mein jaao.
    """
    
    context_so_far = []
    current_question = question
    
    for hop in range(max_hops):
        print(f"\n🔍 Hop {hop+1}: '{current_question}'")
        
        # Step 1: Retrieve
        docs = retriever.invoke(current_question)
        new_context = "\n".join([d.page_content for d in docs])
        context_so_far.append(new_context)
        
        # Step 2: LLM decide kare — answer complete hai ya next hop chahiye?
        decision_prompt = f"""
Original question: {question}
Context collected: {' '.join(context_so_far)}

Kya context se original question ka answer mil sakta hai?
Agar haan: "ANSWER: [your answer]"
Agar nahi, next query: "NEXT_QUERY: [specific follow-up question]"
        """
        
        response = llm.invoke(decision_prompt).content
        
        if response.startswith("ANSWER:"):
            print(f"✅ Answer found in hop {hop+1}!")
            return response.replace("ANSWER:", "").strip()
        elif response.startswith("NEXT_QUERY:"):
            current_question = response.replace("NEXT_QUERY:", "").strip()
    
    return "Maximum hops reached. Incomplete answer."

    Kya hai: Query complexity ke basis par automatically decide karo ki kaunsi retrieval strategy use karni hai — simple queries ke liye fast path, complex ke liye iterative path.

    adaptive_rag.py — LangGraph se Adaptive RAG
    PYTHON
    from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

# State — Saara data yahan store hoga
class RAGState(TypedDict):
    question: str
    documents: list
    generation: str
    query_type: str       # simple / complex / no_retrieval
    hallucination_check: str  # yes / no

# ─── Node 1: Query Classify ────────────────────────
def classify_query(state):
    question = state["question"]
    
    classifier_prompt = f"""
Query classify karo:
- no_retrieval: Simple fact (2+2 = ?)
- simple: Single doc needed
- complex: Multiple docs, multi-hop needed

Query: {question}
Sirf ek word respond karo: no_retrieval / simple / complex
    """
    query_type = llm.invoke(classifier_prompt).content.strip()
    return {"query_type": query_type}

# ─── Node 2a: Simple Retrieve ─────────────────────
def simple_retrieve(state):
    docs = retriever.invoke(state["question"])
    return {"documents": docs}

# ─── Node 2b: Complex Multi-hop Retrieve ──────────
def complex_retrieve(state):
    # Multi-hop retrieval (previous example ki tarah)
    docs = multi_hop_retrieve(state["question"], retriever, llm)
    return {"documents": docs}

# ─── Node 3: Hallucination Check ──────────────────
def check_hallucination(state):
    grading_prompt = f"""
Documents: {state['documents']}
Generated Answer: {state['generation']}

Kya answer documents pe grounded hai? yes / no
    """
    result = llm.invoke(grading_prompt).content.strip()
    return {"hallucination_check": result}

# ─── Graph Build Karo ─────────────────────────────
workflow = StateGraph(RAGState)
workflow.add_node("classify", classify_query)
workflow.add_node("simple_retrieve", simple_retrieve)
workflow.add_node("complex_retrieve", complex_retrieve)
workflow.add_node("hallucination_check", check_hallucination)

# Conditional edges — query type ke basis pe route karo
workflow.add_conditional_edges(
    "classify",
    lambda x: x["query_type"],
    {
        "simple": "simple_retrieve",
        "complex": "complex_retrieve",
        "no_retrieval": END
    }
)
app = workflow.compile()

    6.8 Production RAG — Best Practices
    • Start Simple: Pehle Naive RAG se start karo, phir complexity add karo where needed. Over-engineering avoid karo.
    • Evaluation Pipeline: RAG ko blindly deploy mat karo. RAGAS library se metrics track karo: faithfulness, answer relevancy, context precision.
    • Chunk Quality First: Embedding model se pehle chunking strategy optimize karo. Garbage in = garbage out.
    • Metadata is Gold: Date, source, author, section — yeh saab metadata embedding ke saath store karo. Filtering se accuracy 30%+ improve hoti hai.
    • Hybrid Search Default: Pure semantic ya pure keyword nahi — hybrid ko default banao. Edge cases mein zyaada robust hai.
    • Re-ranking Always: Initial retrieval k=20, phir re-rank karke top-5 LLM ko do. Quality difference bahut zyaada hoti hai.
    • Caching: Popular queries ka answer cache karo. Latency aur cost dono kam hote hain.
    • Monitor Hallucinations: Production mein LLM-as-judge se hallucination rate monitor karo continuously.

    💡 Complex entity relationships → Graph RAG. Simple semantic search → Vector RAG. Cost-benefit analysis karo.

    📋 Chapter Summary — Part 7 Complete Review

    ✅ RAG = Documents index karo → Query pe similar chunks retrieve karo → LLM ko context deke answer generate karo
    ✅ Embeddings: Text ko dense vectors mein convert karo — cosine similarity se similar texts dhundo
    ✅ Vector DBs: ChromaDB (beginner), Pinecone (production), FAISS (offline), Milvus (scale), Weaviate (multimodal)
    ✅ Chunking: Recursive splitting = default choice. Semantic chunking = best quality. Size 300-600 tokens optimal
    ✅ Retrieval: Dense (semantic) + Sparse (BM25) = Hybrid search. Re-ranking se quality improve hoti hai.
    ✅ Advanced RAG: Graph RAG (entities), Agentic RAG (tools), Multi-hop (complex Q), Corrective RAG (self-healing)
    ✅ Evaluation: RAGAS framework se faithfulness, answer_relevancy, context_precision, context_recall measure karo
    ✅ Production: Start simple → Add complexity → Monitor → Optimize. Hybrid search + reranking + RAGAS = production-ready
