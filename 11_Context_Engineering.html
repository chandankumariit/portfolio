CHAPTER 01 · CONTEXT MANAGEMENT
Context Window — LLM ki Short-Term Memory
सन्दर्भ प्रबंधन — वह सब जो LLM देख सकता है
Context window woh magic box hai jiske andar hi LLM sochta hai. Iske bahar kuch bhi nahi — isliye isko engineer karna sabse zaroori skill hai.
🧠
Context Window — Kya Hota Hai?
LLM ek baar mein kitna dekh sakta hai

Socho ek aisi dukaan hai jahan ek intelligent salesman baitha hai. Lekin uski ek limitation hai — woh sirf apni counter ke upar rakhhi cheezein dekh sakta hai. Counter ke upar jo bhi rakha hai — woh sab use yaad hai. Lekin counter se neeche kuch rakha hai toh woh bilkul andhera hai uske liye.

Bilkul aisa hi hota hai LLM (Large Language Model) ke saath. Iska ek Context Window hota hai — ek fixed-size memory area — jiske andar hi woh sab kuch hota hai jo model "dekhta" hai aur sochta hai. Iske bahar kuch nahi hota uske liye.

📖
Definition: Context Window

Context Window woh maximum text/tokens ka amount hota hai jo ek LLM ek API call mein process kar sakta hai. Iske andar — system prompt, chat history, user message, tool results, aur output sab kuch aata hai. Jaise GPT-4 ka 128K tokens hai, Claude ka 200K tokens hai.

📊 DIAGRAM: Context Window ki Anatomy
Context Window ka Size kyun important hai?

Jitna bada context window, utna zyada information ek baar mein process ho sakti hai. Lekin zyada tokens = zyada cost, zyada latency, aur sometimes Lost in the Middle problem — yani model ki attention beech ke tokens par weak ho jaati hai.

⚠️
Common Galti: "Bade window matlab saari problem solve!"

Bahut log sochte hain ki 1M token context window hai toh sab kuch daal do. Lekin reality yeh hai ki zyada irrelevant information = model ka confusion zyada, answer ki quality kam, aur cost bohot zyada. Smart context management zaroor seekho!

Context Engineering kya hai aur kyun zaroori hai?

Context Engineering woh art aur science hai jisme hum decide karte hain ki LLM ke context window mein kya daalna chahiye, kya hataana chahiye, kaise compress karna chahiye — taaki model best possible output de saake, kam tokens mein, kam cost mein, zyada accurately.

🎯 Real-World Analogy

Socho ek surgeon ke operation table ki tarah. Surgeon ke samne sirf woh instruments hone chahiye jo is operation mein kaam aayenge. Agar table par hazaron instruments bhar do, toh surgeon confuse ho jaayega, koi cheez milne mein time lagega. Context Engineering ka kaam hai: sirf sahi tools, sahi jagah, sahi waqt par rakhna.

Token Budgeting — Har Token ka Hisaab
टोकन बजट — हर शब्द की कीमत होती है
Token budgeting woh process hai jisme hum pehle se plan karte hain ki kitne tokens kis kaam ke liye use honge. Jaise ghar ka monthly budget — har cheez ke liye limit set karo.
💰
Token Budgeting — Tokens ka Hisaab Lagaana
Kab kitna token kis jagah use karna chahiye

Token kya hota hai? Ek token roughly 3/4 English word ke barabar hota hai. "Hello world" = 2 tokens. Hindi mein thoda zyada ho sakta hai. Token Budgeting ka matlab hai: total available tokens ko carefully allocate karna — system prompt, history, retrieved docs, user input, aur expected output ke beech mein.

📊 DIAGRAM: Token Budget Allocation (128K window example)
Token Budget kaise set karein? — Step by Step
1
Model ka max context window check karo

Pehle pata karo ki aap jis model ka use kar rahe ho, uska context limit kya hai. GPT-4o = 128K, Claude = 200K, Gemini = 1M. Yahi aapka "total budget" hai.

2
Output ke liye tokens reserve karo (pehle!)

Bhool mat ki output bhi context window consume karta hai! Agar 128K window hai aur aapko 4K output chahiye, toh input ke liye sirf 124K bachta hai. Hamesha pehle output buffer set karo.

3
System prompt ke liye budget fix karo

System prompt stable hota hai — isko concise rakhne ki koshish karo. Typically 1K–8K tokens. Agar bahut lamba hai toh compress karo ya modular banao.

4
Chat history ke liye rolling budget set karo

History hamesha badh'ti rehti hai. Decide karo: max kitni last N turns rakhni hain, ya max K tokens ki history rakhni hai. Baki prune ya summarize kar do.

5
RAG/Tool results ke liye dynamic budget

Retrieval se aaye chunks, tool outputs — yeh dynamically vary karte hain. Inke liye maximum limit set karo aur agar zyada ho toh top-k relevant results hi rakhna.

token_budgeting.py — Practical Token Budget Manager
Python
import tiktoken  # OpenAI ka tokenizer library

class TokenBudgetManager:
    """
    Ek simple token budget manager jo ensure karta hai
    ki hum context window overflow na karein.
    """
    
    def __init__(self, model="gpt-4o", max_context=128000):
        self.model = model
        self.max_context = max_context
        
        # Budget allocation (percentages)
        self.budget = {
            "output_reserve": 4000,    # Output ke liye reserve
            "system_prompt": 6000,    # System prompt max
            "chat_history": 20000,    # History max
            "rag_results": 30000,     # Retrieved docs max
            "user_message": 2000,     # User input max
        }
        
        # Tokenizer initialize karo
        self.encoder = tiktoken.encoding_for_model(model)
    
    def count_tokens(self, text: str) -> int:
        """Kisi bhi text ke tokens count karo"""
        return len(self.encoder.encode(text))
    
    def check_budget(self, component: str, text: str) -> dict:
        """Check karo ki koi component budget mein hai ya nahi"""
        tokens = self.count_tokens(text)
        budget = self.budget.get(component, 5000)
        
        return {
            "component": component,
            "tokens_used": tokens,
            "budget": budget,
            "within_budget": tokens <= budget,
            "overflow": max(0, tokens - budget)
        }
    
    def total_available_input(self) -> int:
        """Kitne tokens input ke liye available hain"""
        return self.max_context - self.budget["output_reserve"]
    
    def build_context(self, system_prompt, history, rag_docs, user_msg):
        """Smart context build karo budget ke andar reh kar"""
        context = {}
        used = 0
        
        # 1. System prompt (fixed)
        sys_tokens = self.count_tokens(system_prompt)
        if sys_tokens > self.budget["system_prompt"]:
            print(f"⚠️ System prompt too long: {sys_tokens} tokens!")
        context["system"] = system_prompt[::-1][::-1]  # use as is
        used += sys_tokens
        
        # 2. RAG results (top K jo budget mein fit ho)
        rag_budget = self.budget["rag_results"]
        selected_docs = []
        for doc in rag_docs:  # Already ranked by relevance
            doc_tokens = self.count_tokens(doc)
            if rag_budget >= doc_tokens:
                selected_docs.append(doc)
                rag_budget -= doc_tokens
                used += doc_tokens
            else:
                break  # Budget khatam, aur mat lao
        context["rag"] = selected_docs
        
        # 3. Chat history (recent first, budget ke andar)
        hist_budget = self.budget["chat_history"]
        selected_history = []
        for turn in reversed(history):  # Recent turns pehle
            turn_tokens = self.count_tokens(str(turn))
            if hist_budget >= turn_tokens:
                selected_history.insert(0, turn)
                hist_budget -= turn_tokens
                used += turn_tokens
            else:
                break
        context["history"] = selected_history
        
        # 4. User message
        context["user"] = user_msg
        used += self.count_tokens(user_msg)
        
        print(f"✅ Total tokens used: {used}/{self.max_context}")
        return context

# Usage example:
manager = TokenBudgetManager(max_context=128000)
result = manager.check_budget("system_prompt", "You are a helpful AI...")
print(result)
# Output: {'component': 'system_prompt', 'tokens_used': 5, 'budget': 6000, 'within_budget': True, 'overflow': 0}

Pro Tip: Tiktoken vs Transformers Tokenizer

OpenAI models ke liye tiktoken use karo (exact count). Anthropic Claude ke liye anthropic.count_tokens() API method use karo. Hugging Face models ke liye AutoTokenizer use karo. Approximate estimate ke liye: 1 token ≈ 4 characters English mein, ya "word count × 1.3".

Context Pruning — Kya Hataana Chahiye?
सन्दर्भ छँटाई — अनावश्यक जानकारी हटाना
Context pruning woh process hai jisme hum context se woh content hata dete hain jo current task ke liye irrelevant ya redundant hai — taaki model ki quality improve ho aur cost kam ho.
✂️
Context Pruning — Kya Hataana Chahiye?
Pruning strategies — smart removal

Context Pruning seedha simple hai: context se unnecessary cheezein nikaal do. Lekin tricky part yeh hai — kya zaroori hai aur kya nahi, yeh decide karna art hai. Galat cheez hatao toh model ka performance gir jaata hai. Sahi cheez hatao toh cost aur speed dono improve hote hain.

📊 FLOWCHART: Context Pruning Decision Process
Pruning ke Types — Kaun si Strategy Kab Use Karein?
📅 Recency-Based Pruning (Time-based)

Kya hai: Sirf last N turns ya last K minutes ki conversation rakhna. Purani baatein hata do.

Kab use karein: Jab conversation linear hai aur old context rarely refer hoti ho. Customer support chatbots.

Drawback: Agar user 50 turns baad phir kisi purani baat ka reference de, toh model confuse ho sakta hai.

def recency_prune(history, max_turns=10):
    # Sirf last max_turns turns rakhein
    return history[-max_turns:]  # Simple!
🎯 Relevance-Based Pruning (Semantic)

Kya hai: Har context item ko current query ke saath semantically compare karo (embedding similarity), aur jo sirf kuch threshold se upar ho, unhe hi rakhna.

Kab use karein: Long conversations jahan sirf thode hi turns relevant hote hain. Research agents.

from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

def relevance_prune(history, query_embedding, embeddings, threshold=0.7):
    """Sirf relevant turns rakhna"""
    kept = []
    for i, turn in enumerate(history):
        similarity = cosine_similarity(
            [query_embedding], [embeddings[i]]
        )[0][0]
        if similarity >= threshold:
            kept.append(turn)
    return kept
🏷️ Importance Scoring Pruning

Kya hai: Har context item ko ek "importance score" assign karna based on multiple factors — recency, frequency of reference, explicit user marking, etc. Score ke basis par prune karo.

Kab use karein: Sophisticated long-running agents jaise coding assistants, research assistants.

def importance_score(turn, position, total, query_similarity):
    recency = position / total       # 0 (old) to 1 (new)
    relevance = query_similarity     # 0 to 1
    is_user = 0.3 if turn['role'] == 'user' else 0.1
    return 0.4*recency + 0.5*relevance + 0.1*is_user
🚨
Pruning mein Common Mistakes — Inse Bachna!

1. System prompt ko kabhi prune mat karo — yeh always rakhna chahiye.
2. Current user message ko kabhi prune mat karo — latest question toh chahiye hi.
3. Tool call results ko aggressively prune karna — agent ke decision making ko hurt kar sakta hai.
4. Bina testing ke prune karna — pehle evaluate karo ki pruning se quality giri toh nahi.

Context Prioritization — Kise Pehle Rakhein?
प्राथमिकता — कौन सी जानकारी पहले आनी चाहिए
LLMs mein "Lost in the Middle" problem hota hai — model ke beginning aur end mein rakhi information zyada influence karti hai. Prioritization isi problem ka solution hai.
🎯
Context Prioritization — Sahi Order Mein Rakhna
Lost in the Middle problem ko solve karo

Research se pata chala hai ki LLMs context ke shuru (beginning) aur ant (end) ki information ko zyada "attend" karte hain. Beech mein rakhi information often "lost" ho jaati hai — model usse overlook kar deta hai. Is phenomenon ko Lost in the Middle problem kehte hain.

📊 DIAGRAM: Lost in the Middle Problem
Context ka Sahi Order — Priority Stack

Isi problem ko solve karne ke liye ek tested pattern hai: Priority Stack — sabse important information pehle (beginning) aur akhir (end) mein rakhna.

📊 DIAGRAM: Recommended Context Order (Priority Stack)
💡
Remember This: "Bread Technique"

Context prioritization ko "Sandwich" ki tarah socho: sabse important cheez pehle (upar ki roti) rakhna, supporting content beech mein, aur current task/question akhir mein (neeche ki roti). Model ko beginning aur end yaad rehta hai — wahi "roti" hai jo usse pata rehti hai!

Practical Prioritization Code
context_prioritizer.py
Python
def build_prioritized_context(
    system_prompt: str,
    rag_chunks: list[str],      # Already ranked by relevance
    tool_results: list[str],
    chat_history: list[dict],
    current_message: str
) -> list[dict]:
    """
    Priority Stack build karo — important cheezein pehle aur akhir mein.
    Lost-in-the-Middle problem minimize karo.
    """
    messages = []
    
    # ① System prompt — ALWAYS first (beginning attention)
    messages.append({
        "role": "system",
        "content": system_prompt
    })
    
    # ② RAG chunks — most relevant first (near beginning)
    if rag_chunks:
        rag_content = "\n\n".join([
            f"[Retrieved Document {i+1}]\n{chunk}"
            for i, chunk in enumerate(rag_chunks)
        ])
        messages.append({
            "role": "user",
            "content": f"Reference documents:\n{rag_content}"
        })
        messages.append({
            "role": "assistant",
            "content": "I've reviewed the reference documents."
        })
    
    # ③ Tool results
    for result in tool_results:
        messages.append({"role": "tool", "content": result})
    
    # ④ Chat history (middle — least critical for attention)
    messages.extend(chat_history)
    
    # ⑤ Current message — ALWAYS last (end attention)
    messages.append({
        "role": "user",
        "content": current_message
    })
    
    return messages

# Usage:
context = build_prioritized_context(
    system_prompt="You are an expert Python developer...",
    rag_chunks=["Python async functions...", "Event loop basics..."],
    tool_results=["Search result: asyncio docs..."],
    chat_history=[{"role": "user", "content": "What is asyncio?"}],
    current_message="Give me a working example of async/await."
)

Context Caching — Dobara Kaam Na Karein
कैशिंग — एक बार process करो, बार-बार reuse karo
Context caching ek powerful optimization hai jisme ek baar processed context ko reuse kiya jaata hai, naye calls mein phir se process karne ki zaroorat nahi. Cost 90% tak kam ho sakti hai!
⚡
Context Caching — Speed aur Cost ka Ek Saath Solution
90% cost reduction possible hai caching se

Socho ek teacher hai jo har student ke saath same textbook chapters explain karta hai — baar baar wahi kaam. Context Caching mein, pehli baar processed content ka result "save" kar lete hain (cache), aur agali baar wahi content phir se process karne ki zaroorat nahi — sirf cached version reuse kar lo.

Anthropic Claude ka Prompt Caching feature, OpenAI ka Prompt Caching — yeh dono same prefix ko cache karte hain. Iska matlab: ek lambi system prompt ko ek baar process karo, baaki saare requests mein sirf cached version use karo.

❌ Without Caching
  • Har request mein poora context re-process
  • 100 requests × 10K tokens = 1M input tokens
  • Full cost har baar
  • Higher latency (reprocessing time)
  • $5/1M tokens × 1M = $5.00
✅ With Caching
  • Prefix ek baar process, 100× reuse
  • 1 full + 99 × small (dynamic part only)
  • Cache hit: ~10% ka cost
  • Lower latency (skip reprocessing)
  • Savings: up to 90% cost reduction!

Context Caching — Code Implementation
context_caching_anthropic.py — Claude Prompt Caching
Python
import anthropic

client = anthropic.Anthropic()

# Long system prompt / context jo cache karna chahte hain
LONG_SYSTEM_CONTEXT = """You are an expert Python developer...
[Very long instructions - 8000 tokens worth of content]
"""

def ask_with_caching(user_question: str) -> str:
    """
    cache_control ke saath Anthropic ka prompt caching use karo.
    Pehli request: cache WRITE (full cost)
    Agle requests: cache READ (10% cost!)
    """
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=1024,
        system=[
            {
                "type": "text",
                "text": LONG_SYSTEM_CONTEXT,
                "cache_control": {"type": "ephemeral"}
                # ↑ Yeh line cache enable karti hai!
            }
        ],
        messages=[
            {"role": "user", "content": user_question}
        ]
    )
    
    # Cache usage stats check karo
    usage = response.usage
    print(f"📊 Cache Stats:")
    print(f"  Input tokens: {usage.input_tokens}")
    print(f"  Cache write: {usage.cache_creation_input_tokens}")
    print(f"  Cache read: {usage.cache_read_input_tokens}")
    
    if usage.cache_read_input_tokens > 0:
        print("  ✅ Cache HIT — Cost saved!")
    else:
        print("  📝 Cache MISS — First request, cache created")
    
    return response.content[0].text

# First call: Cache write (poora cost)
ans1 = ask_with_caching("Python mein decorators kya hote hain?")

# Second call onward: Cache hit (90% cost saved!)
ans2 = ask_with_caching("Generators aur iterators mein kya fark hai?")
ans3 = ask_with_caching("Async/await ka use kab karein?")
💡
Context Caching Best Practices

1. Static content pehle rakhna — system prompt, long docs — yeh sab cache hone chahiye.
2. Dynamic content baad mein — user messages, real-time data — yeh cache nahi hota.
3. Cache TTL (Time-to-Live) — Anthropic mein 5 minutes, OpenAI mein 1 hour. Is window ke andar reuse karo.
4. Minimum tokens — Claude ke liye minimum 1024 tokens chahiye cache ke liye. Chhoti content cache nahi hoti.

Compression — Context ko Chhota Banana
ज्यादा information कम tokens में

Context Compression ka matlab hai — existing content ko is tarah rewrite ya restructure karna ki same information kam tokens mein fit ho jaaye. Sochiye ek telegram message — seedha, concise, koi faaltu shabad nahi — lekin full meaning preserved.

Compression ke Types
📝 Selective Compression — Sirf Verbose Parts Compress Karo

Chat history mein verbose user messages aur long AI responses ko compress karo. Tool results mein se sirf key numbers aur facts rakhna. Code blocks mein comments aur whitespace reduce karo (for context — production mein mat karo!).

async def compress_turn(turn: dict, llm_client) -> dict:
    """LLM se ek conversation turn compress karao"""
    prompt = f"""Compress karo yeh message to 30% of original length.
Preserve: key facts, decisions, important numbers.
Remove: pleasantries, repetition, verbose explanations.

Original: {turn['content']}

Compressed version (ONLY the compressed text, nothing else):"""
    
    compressed = await llm_client.generate(prompt)
    return {
        "role": turn["role"],
        "content": compressed,
        "_compressed": True  # Track karo
    }
🔑 Key-Value Compression — Structured Format

Natural language conversation ko structured key-value format mein convert karna. Example: "The user mentioned they prefer Python over JavaScript and have 3 years of experience" → lang_pref: Python, exp_years: 3

def extract_key_facts(conversation_history: list) -> str:
    """Conversation se key facts extract karo JSON format mein"""
    import json
    
    facts = {
        "user_preferences": [],
        "decisions_made": [],
        "key_numbers": {},
        "important_context": []
    }
    # LLM se extract karao ya regex se parse karo
    # Example result:
    facts["user_preferences"] = ["Python", "FastAPI", "PostgreSQL"]
    facts["decisions_made"] = ["Use microservices architecture"]
    
    # JSON compact format — ~60% space savings!
    return json.dumps(facts, separators=(',', ':'))
🧠 LLM-Powered Compression

Ek cheaper/faster LLM (jaise GPT-3.5 ya Haiku) use karke expensive model ka context compress karo. Yeh "compression model" faster hota hai aur sirf summarization karta hai — main task nahi.

async def smart_compress_history(history, cheap_model="gpt-3.5-turbo"):
    """Cheap model se compress karao, expensive model se use karo"""
    old_turns = history[:-5]  # Last 5 rakhna, baki compress
    recent_turns = history[-5:]
    
    if not old_turns:
        return history
    
    compression_prompt = f"""
    Compress this conversation history into a brief summary (max 200 words).
    Focus on: decisions, facts, user preferences, important context.
    
    History: {old_turns}
    """
    
    summary = await call_cheap_llm(compression_prompt, model=cheap_model)
    
    compressed_context = [{
        "role": "system",
        "content": f"[Earlier conversation summary]: {summary}"
    }] + recent_turns
    
    return compressed_context
⚠️
Compression ke Risks

Har compression mein information loss ka risk hota hai. Aggressive compression quality hurt kar sakta hai. Always evaluate: compress karne ke baad model ki response quality verify karo. Critical information (passwords, exact numbers, legal terms) kabhi compress mat karo — always verbatim rakhna.

Summarization — Lambi Baat Chhoti Karna
बड़े context को छोटे summary में बदलना

Summarization aur compression mein fark hai: Compression original text ko reformulate karta hai, jabki Summarization ek new, concise representation generate karta hai jo original content ke key points capture karta hai. AI agents ke liye yeh ek critical technique hai.

📖 Analogy

Socho ek book ka synopsis. Poori 400-page novel ki jagah, ek 2-page summary di jaaye jisme plot, characters, aur key themes ho. Reader ko poori novel ki core idea mil jaati hai — bina 400 pages padhay. Exactly aisa hi context summarization karta hai conversation history ke liye.

Summarization Strategies — Kaunsi Strategy Kab?
StrategyKya hota haiBest Use CaseTradeoff
Rolling Summary Har N turns baad, ab tak ki conversation ka summary banao aur old turns replace karo Customer support, long chatbots Some detail loss, but manageable
Hierarchical Summary Recent = full detail, old = paragraph, very old = bullet points Research agents, coding assistants More complex to implement
Topic-Based Summary Topics ke basis par summarize karo, not time-based Multi-topic conversations Requires topic detection
Structured Summary Fixed JSON/YAML format mein summary: decisions, facts, preferences Agentic workflows Rigid format, some nuance lost
rolling_summarizer.py — Production-Grade Summarization
Python
from dataclasses import dataclass
from typing import Optional
import json

@dataclass
class ConversationMemory:
    """Structured conversation memory jo summarization track kare"""
    rolling_summary: Optional[str] = None
    recent_turns: list = None
    key_facts: dict = None
    turn_count: int = 0
    
    def __post_init__(self):
        if self.recent_turns is None:
            self.recent_turns = []
        if self.key_facts is None:
            self.key_facts = {}

class RollingSummarizer:
    """
    Rolling summarization manager.
    
    Strategy:
    - Last 6 turns: full detail rakhna (verbatim)
    - Older turns: progressive summarization
    - Key facts: always extracted aur maintained
    """
    
    def __init__(self, llm_client, window_size=6, summarize_every=10):
        self.llm = llm_client
        self.window_size = window_size  # Recent turns full detail
        self.summarize_every = summarize_every  # Kitne turns baad summarize
        self.memory = ConversationMemory()
    
    async def add_turn(self, role: str, content: str):
        """Naya turn add karo aur summarize karo agar zaroorat ho"""
        self.memory.recent_turns.append({
            "role": role,
            "content": content
        })
        self.memory.turn_count += 1
        
        # Har N turns baad summarize karo
        if self.memory.turn_count % self.summarize_every == 0:
            await self._summarize_old_turns()
    
    async def _summarize_old_turns(self):
        """Purane turns ko summarize karo"""
        if len(self.memory.recent_turns) <= self.window_size:
            return
        
        # Old turns = window ke bahar wale
        old_turns = self.memory.recent_turns[:-self.window_size]
        
        # Context build karo summary ke liye
        prev_summary = self.memory.rolling_summary or "(No previous summary)"
        turns_text = "\n".join([
            f"{t['role'].upper()}: {t['content']}"
            for t in old_turns
        ])
        
        prompt = f"""Update the conversation summary with new turns.

Previous Summary:
{prev_summary}

New Turns to Incorporate:
{turns_text}

Create an updated summary that:
1. Key decisions aur facts preserve karo
2. User preferences note karo  
3. Any errors ya corrections note karo
4. Max 300 words mein rakhna

Updated Summary:"""
        
        self.memory.rolling_summary = await self.llm.generate(prompt)
        
        # Only recent window rakhna
        self.memory.recent_turns = self.memory.recent_turns[-self.window_size:]
        print(f"✅ Summarized {len(old_turns)} old turns. Memory compact!")
    
    def get_context(self) -> list[dict]:
        """Current optimized context return karo"""
        context = []
        
        # Rolling summary (agar hai toh)
        if self.memory.rolling_summary:
            context.append({
                "role": "system",
                "content": f"[Conversation History Summary]:\n{self.memory.rolling_summary}"
            })
        
        # Recent full turns
        context.extend(self.memory.recent_turns)
        return context

import chromadb
from openai import OpenAI
import json
from datetime import datetime

class RetrievalMemory:
    """
    Vector database pe based agent memory system.
    Isme unlimited memories store ho sakti hain,
    aur sirf relevant memories context mein aati hain.
    """
    
    def __init__(self, user_id: str):
        self.user_id = user_id
        self.openai = OpenAI()
        
        # ChromaDB client setup
        self.client = chromadb.PersistentClient(path=f"./memory_{user_id}")
        self.collection = self.client.get_or_create_collection(
            name="agent_memories",
            metadata={"hnsw:space": "cosine"}
        )
    
    def _get_embedding(self, text: str) -> list[float]:
        """Text ka embedding banana"""
        response = self.openai.embeddings.create(
            model="text-embedding-3-small",
            input=text
        )
        return response.data[0].embedding
    
    def store(self, content: str, memory_type: str = "general"):
        """Memory store karo vector DB mein"""
        embedding = self._get_embedding(content)
        memory_id = f"{self.user_id}_{datetime.now().timestamp()}"
        
        self.collection.add(
            ids=[memory_id],
            embeddings=[embedding],
            documents=[content],
            metadatas=[{
                "user_id": self.user_id,
                "type": memory_type,
                "timestamp": datetime.now().isoformat()
            }]
        )
        print(f"💾 Memory stored: '{content[:50]}...'")
    
    def retrieve(self, query: str, top_k: int = 5) -> list[str]:
        """Query ke relevant memories retrieve karo"""
        query_embedding = self._get_embedding(query)
        
        results = self.collection.query(
            query_embeddings=[query_embedding],
            n_results=min(top_k, self.collection.count()),
            where={"user_id": self.user_id}
        )
        
        memories = results['documents'][0] if results['documents'] else []
        print(f"🔍 Retrieved {len(memories)} relevant memories")
        return memories
    
    def build_context_with_memory(self, query: str, system_prompt: str) -> list:
        """Context banao with relevant memories injected"""
        relevant_memories = self.retrieve(query)
        
        if relevant_memories:
            memory_text = "\n".join([f"- {m}" for m in relevant_memories])
            enhanced_system = f"""{system_prompt}

[Relevant memories about this user]:
{memory_text}
"""
        else:
            enhanced_system = system_prompt
        
        return [
            {"role": "system", "content": enhanced_system},
            {"role": "user", "content": query}
        ]

# Next session mein:

    query="How do I handle rate limiting?",
    system_prompt="You are a helpful coding assistant."
)

from dataclasses import dataclass, asdict
import json
import sqlite3
from datetime import datetime
from typing import Optional, Any

@dataclass
class UserMemory:
    """User ki structured persistent memory"""
    user_id: str
    name: Optional[str] = None
    preferences: dict = None        # Programming language, style, etc.
    projects: list = None           # Active projects
    expertise_level: str = "beginner" # beginner/intermediate/expert
    interaction_count: int = 0
    last_seen: Optional[str] = None
    custom_facts: list = None       # Agent ne jo facts note kiye
    
    def __post_init__(self):
        if self.preferences is None: self.preferences = {}
        if self.projects is None: self.projects = []
        if self.custom_facts is None: self.custom_facts = []

class PersistentMemoryManager:
    """SQLite-backed persistent memory manager"""
    
    def __init__(self, db_path: str = "agent_memory.db"):
        self.conn = sqlite3.connect(db_path, check_same_thread=False)
        self._init_db()
    
    def _init_db(self):
        self.conn.execute("""
            CREATE TABLE IF NOT EXISTS user_memory (
                user_id TEXT PRIMARY KEY,
                memory_json TEXT NOT NULL,
                updated_at TEXT NOT NULL
            )
        """)
        self.conn.commit()
    
    def load(self, user_id: str) -> UserMemory:
        """User ki memory load karo"""
        cursor = self.conn.execute(
            "SELECT memory_json FROM user_memory WHERE user_id = ?",
            (user_id,)
        )
        row = cursor.fetchone()
        
        if row:
            data = json.loads(row[0])
            return UserMemory(**data)
        return UserMemory(user_id=user_id)  # New user
    
    def save(self, memory: UserMemory):
        """Memory save karo (upsert)"""
        memory.last_seen = datetime.now().isoformat()
        memory.interaction_count += 1
        
        self.conn.execute("""
            INSERT OR REPLACE INTO user_memory 
            VALUES (?, ?, ?)
        """, (
            memory.user_id,
            json.dumps(asdict(memory)),
            memory.last_seen
        ))
        self.conn.commit()
    
    def to_context_string(self, memory: UserMemory) -> str:
        """Memory ko context-friendly string mein convert karo"""
        parts = []
        if memory.name:
            parts.append(f"User ka naam: {memory.name}")
        if memory.expertise_level:
            parts.append(f"Expertise level: {memory.expertise_level}")
        if memory.preferences:
            parts.append(f"Preferences: {json.dumps(memory.preferences)}")
        if memory.projects:
            parts.append(f"Active projects: {', '.join(memory.projects)}")
        if memory.custom_facts:
            parts.append("Known facts:")
            parts.extend([f"  - {f}" for f in memory.custom_facts[-5:]])
        
        return "\n".join(parts) if parts else "New user — no history"

Sliding Window — Types aur Variants
🔢 Fixed-Count Window (N last turns)

Simplest approach: hamesha sirf last N turns rakhna. Jaise "last 10 messages". Iska implementation trivial hai.

def fixed_window(history, n=10):
    return history[-n:]  # Last N turns

# Pro: Simple | Con: Doesn't account for token length
⚖️ Token-Budget Window (K tokens max)

N turns nahi, balki maximum K tokens ki history rakhna. Zyada balanced kyunki kuch turns bahut lambe hote hain.

def token_budget_window(history, max_tokens=8000, encoder=None):
    """Recent turns rakhna token budget ke andar"""
    result = []
    total = 0
    for turn in reversed(history):  # Recent pehle
        tokens = len(encoder.encode(str(turn))) if encoder else len(str(turn)) // 4
        if total + tokens <= max_tokens:
            result.insert(0, turn)
            total += tokens
        else:
            break
    return result

# Pro: Precise token control | Con: Slightly more complex
🧩 Hybrid Window (Summary + Recent Full)

Best of both worlds: Old turns ka summary + Recent turns full verbatim. Yeh production mein sabse common pattern hai.

def hybrid_window(history, recent_count=6, summary=None):
    """Summary + recent full turns"""
    context = []
    if summary:
        context.append({
            "role": "system",
            "content": f"Previous conversation: {summary}"
        })
    context.extend(history[-recent_count:])
    return context

# Best balance: Cost efficient + Quality maintained

class ContextAwareAgent:
    """
    Production-grade agent jo context engineering
    ke saath kaam karta hai.
    """
    
    def __init__(self, user_id: str):
        self.budget = TokenBudgetManager(max_context=128000)
        self.memory = RetrievalMemory(user_id)
        self.persistent = PersistentMemoryManager()
        self.summarizer = RollingSummarizer(llm_client=self)
        self.system_prompt = "You are a helpful AI assistant."
    
    async def respond(self, user_message: str) -> str:
        """Main agent loop with full context engineering"""
        
        # Step 1: Persistent memory load karo
        user_mem = self.persistent.load(self.memory.user_id)
        
        # Step 2: Relevant memories retrieve karo
        retrieved = self.memory.retrieve(user_message, top_k=5)
        
        # Step 3: Optimized context build karo
        context = self.budget.build_context(
            system_prompt=self._enhance_system_prompt(user_mem),
            history=self.summarizer.get_context(),
            rag_docs=retrieved,
            user_msg=user_message
        )
        
        # Step 4: LLM call
        response = await self._call_llm(context)
        
        # Step 5: Memory update karo
        await self.summarizer.add_turn("user", user_message)
        await self.summarizer.add_turn("assistant", response)
        self.memory.store(
            f"User asked: {user_message[:100]}",
            memory_type="episodic"
        )
        self.persistent.save(user_mem)
        
        return response
    
    def _enhance_system_prompt(self, user_memory: UserMemory) -> str:
        """System prompt ko user memory se augment karo"""
        memory_context = self.persistent.to_context_string(user_memory)
        return f"""{self.system_prompt}

[User Profile]:
{memory_context}"""

Interview Questions — Context Engineering
Q1. Context window aur working memory mein kya fark hai? Production mein kaise manage karein?

Context window LLM ki immediate processing capacity hai — fixed size (128K tokens etc.). Working memory ek broader concept hai jo multiple layers par span karta hai: in-context, cache, vector DB, aur persistent store. Production mein: Token budget manager use karo, rolling summarization implement karo, aur retrieval memory se sirf relevant info inject karo.

Q2. "Lost in the Middle" problem kya hai aur isse kaise solve karein?

Research ne show kiya hai ki LLMs beginning aur end ki information ko zyada attend karte hain, middle mein rakhi info often overlooked ho jaati hai. Solution: Priority Stack — most critical info beginning mein rakho (system prompt, core constraints), current task end mein rakho, supporting context middle mein. Repetition of critical constraints bhi help karta hai.

Q3. Context caching se cost kitna bachta hai aur kaise implement karein?

Anthropic ka Prompt Caching cache read tokens ko 90% discount deta hai (0.3$/1M instead of 3$/1M). Implementation: cache_control: "ephemeral" flag use karo static prefix (system prompt + docs) par. Cache TTL 5 minutes hai — is window mein saari requests cache hit karein. Minimum 1024 tokens cache ke liye required. Real savings: 100 requests × 10K token prefix = 1M input tokens. Without cache: $3.00. With cache: ~$0.30 + one-time write. 90% savings!

Q4. Retrieval memory aur traditional RAG mein kya fark hai?

Traditional RAG documents retrieve karta hai, Retrieval Memory conversation history aur agent observations ko retrieve karta hai. RAG = knowledge base se external docs. Retrieval Memory = agent ki apni past experiences, user preferences, session history — jo vector DB mein store hoti hain aur semantically retrieve hoti hain. Dono combine karna best approach hai.

Q5. Token budgeting implement karne ke steps kya hain?

1) Model ka max context window pata karo. 2) Output buffer reserve karo pehle (typically 2K-8K tokens). 3) System prompt budget fix karo (1K-8K). 4) Remaining tokens proportionally allocate karo: RAG (30-40%), History (15-20%), User Input (2-5%). 5) Tiktoken ya model-specific API se actual counts measure karo. 6) Overflow pe graceful degradation implement karo (prune history, reduce RAG chunks).

Sliding Windowसरकती खिड़कीFixed-size window of recent turns. Jaise conversation aage badhe, window bhi slide karti hai.
Rolling Summaryचलता सारांशHar N turns baad purani conversation ka summary banao, old turns replace karo. Running memory technique.
Retrieval Memoryपुनःप्राप्ति स्मृतिVector DB mein stored memories jo query ke saath semantically match karke retrieve hoti hain.
Persistent Memoryस्थायी स्मृतिUser information jo sessions ke paas bhi SQL/NoSQL mein stored rehti hai. Personalization enable karta hai.
Lost in the Middleबीच में खोनाResearch finding: LLMs context ke beginning aur end par zyada focus karte hain, middle often overlook hota hai.
Hierarchical Memoryश्रेणीबद्ध स्मृतिHot/Warm/Cool/Cold memory levels. In-context → Cache → Vector DB → SQL — speed vs capacity tradeoff.
Attention Optimizationध्यान अनुकूलनContext ko aise structure karna ki model ka attention mechanism sahi cheezein focus kare.
Compressionसंपीड़नSame information kam tokens mein represent karna. Key-value format, LLM-powered compression, selective compression.
Cache TTLकैश अवधिTime-to-Live — kitni der tak cache valid rahega. Anthropic: 5 min, OpenAI: 1 hour.
